### [Model Support (DeepSeek)] — PR #24231: New GGML_OP_LIGHTNING_INDEXER that implements DeepSeek V3.2/V4 lightning indexer

**Merged:** 2026-07-11 | **Author:** @fairydreaming

**Summary:** Adds a new `GGML_OP_LIGHTNING_INDEXER` operation that implements the DeepSeek V3.2/V4 lightning indexer as a fused GGML operation. The lightning indexer computes attention-like scores between queries and a compressed KV cache using ReLU-activated dot products weighted by learned head weights. By fusing the einsum matrix multiplication, ReLU activation, weighted sum, and head reduction into a single op, the compute buffer size is reduced by 64× — from 168,368 MiB down to just 5,808 MiB for DeepSeek V3.2 at full 163,840 context. Performance is slightly improved on CPU (pp512: 25.21→25.51 t/s, tg32: 6.29→6.52 t/s). This is a critical enabler for running DeepSeek V3.2/V4 models with full context windows without running out of memory.

**Affected areas:** DeepSeek V3.2/V4, GGML kernel, lightning indexer, compute buffer optimization

### [Bug Fix (DeepSeek V4)] — PR #25521: DeepseekV4: clear cache only for seq rather than full

**Merged:** 2026-07-11 | **Author:** @am17an

**Summary:** Fixes a DeepSeek V4 cache amnesia bug (issue #25171) where the entire compressed KV cache was cleared on every cache clear operation instead of only clearing the cache for the specific sequence ID. In multi-sequence scenarios (e.g., continuous batching, conversation branching), this caused other sequences to lose their cached context, effectively giving the model "amnesia." The fix changes the cache clear logic to target individual seq_ids rather than performing a full wipe. This is essential for DeepSeek V4 users running the server with multiple concurrent slots or any multi-sequence workflow.

**Affected areas:** DeepSeek V4, compressed attention (CSA/HCA), KV cache management

### [Server/API] — PR #25526: server: allow stream for exec_shell_command

**Merged:** 2026-07-11 | **Author:** @ngxson

**Summary:** Enables streaming output (`"stream": true`) for the `exec_shell_command` tool in llama-server's OpenAI-compatible tools API. Previously, shell command execution was always buffered — the full output was returned only after the command completed. With this change, users can receive command output token-by-token as it's produced, which is useful for long-running commands (e.g., build processes, data processing scripts) where users want to see progress in real-time. The feature is enabled by setting `stream: true` in the tool call parameters. Documentation has been updated accordingly.

**Affected areas:** Server, tools API, exec_shell_command, streaming

### [Server/API] — PR #25541: server: refactor server_stream

**Merged:** 2026-07-11 | **Author:** @ngxson

**Summary:** Refactors the server's stream replay (SSE pipe) mechanism to make the code more self-contained and thread-safe. The `spipe` (stream pipe) and drain logic has been extracted from `server_http_res` into a new `server_res_spipe` class that extends `server_http_res` and uses a tee-style pipe pattern via `next()`. This is a code quality improvement that makes the streaming infrastructure cleaner and safer for concurrent access, with no user-facing behavioral changes.

**Affected areas:** Server, SSE streaming, code refactoring, thread safety

### [Bug Fix (CUDA)] — PR #25157: cuda: fix crash when querying memory on device with no free memory

**Merged:** 2026-07-11 | **Author:** @cphlipot

**Summary:** Fixes a crash in the CUDA backend where `cudaMemGetInfo()` itself could fatally fail when a CUDA device has no available memory — the CUDA OOM condition prevents even querying memory stats. The fix makes the memory query error non-fatal: if the query fails, the device is reported as having zero total/free memory, which causes the model fit algorithm to skip that device rather than crash. This also prevents crashes in CUDA-enabled builds when the user explicitly passes `--dev none`. Partial fix for issue #25145 (note: mmproj may still need `--no-mmproj-offload` to fully prevent CUDA allocations).

**Affected areas:** CUDA backend, memory management, error handling, server startup

### [Bug Fix] — PR #24926: Fix conditional to display 'LLAMA_SPLIT_MODE_TENSOR not implemented for architecture' message

**Merged:** 2026-07-11 | **Author:** @kdkd

**Summary:** Fixes a broken early-exit check that was supposed to prevent models with unsupported architectures from crashing with an opaque assert when using `--split-mode tensor` (`-sm tensor`). The original code checked `devices[0].is_meta` to determine if a GPU backend was in use, but `devices` had not been populated yet at that point in model loading, so the check was always skipped. The fix simplifies the conditional to just check `params.split_mode == LLAMA_SPLIT_MODE_TENSOR && !llm_arch_supports_sm_tensor(arch)`, which correctly catches unsupported architectures like GLM-5.2 with DeepSeek Sparse Attention (glm-dsa) and exits cleanly in ~3 seconds instead of crashing 90+ seconds later with a confusing assertion error.

**Affected areas:** Model loading, split mode, error handling, GLM-5.2 DSA
