### [Backend] — PR #22930: vulkan: prefer host-visible memory buffers on UMA devices

**Merged:** 2026-06-16 | **Author:** @winstonma

**Summary:** Modifies the Vulkan backend to allocate device-local buffers with the `VK_MEMORY_PROPERTY_HOST_VISIBLE_BIT` flag on UMA (Unified Memory Architecture) devices, where CPU and GPU share the same physical memory. Previously, even on UMA hardware, the backend allocated without host visibility, forcing an unnecessary staging copy whenever tensor data needed to be read back to the host (e.g., during loss evaluation in training). This change eliminates that copy overhead on integrated GPUs and APUs such as AMD Radeon 890M. This is a correctness/performance fix for UMA Vulkan devices — no user-facing flags are changed.

**Affected areas:** Vulkan backend, UMA memory allocation

### [Backend] — PR #24425: vulkan: add col2im_1d op

**Merged:** 2026-06-16 | **Author:** @ServeurpersoCom

**Summary:** Adds the `GGML_OP_COL2IM_1D` operation to the Vulkan backend, implementing a gather shader that maps one invocation per output element, reading only the `ceil(K/stride)` useful columns. Supports F32, F16, and BF16 types (BF16 converts through `bf16_to_fp32` for devices without native BF16). An optimization commit improves performance by walking the bounded `t_in` range instead of scanning all K with modulo operations, achieving up to ~7× speedup on cache-resident vocoder stage shapes. This completes Vulkan coverage for the col2im_1d operation previously added for CPU, benefiting audio vocoder upsampling pipelines.

**Affected areas:** Vulkan backend, shader pipeline, audio processing

### [Server/API] — PR #24536: spec: add spec metrics mean acceptance length and acceptance rate per position

**Merged:** 2026-06-16 | **Author:** @ruixiang63

**Summary:** Adds `mean_acceptance_length` and `acceptance_rate_per_position` metrics for speculative decoding, following vLLM's spec metrics definitions. `mean_acceptance_length = 1 + accepted_tokens / draft_verification_steps` and `acceptance_rate_per_position[i]` tracks acceptance at each draft position. This helps users tune parameters like `--spec-draft-n-max` and `--spec-draft-p-min` by providing visibility into how well each draft position is being accepted across requests.
