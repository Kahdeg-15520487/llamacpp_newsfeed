### [Performance] — PR #18039: [Speculative decoding] feat: add EAGLE3 speculative decoding support

**Merged:** 2026-06-12 | **Author:** @ruixiang63

**Summary:** This is the landmark integration of EAGLE3 speculative decoding into llama.cpp, a collaboration between NVIDIA and GGML teams. EAGLE3 uses an encoder-decoder architecture to extract features from the target model at specific layers, compress them through a feature fusion layer, and generate draft tokens with a single-layer decoder — achieving 2–3× speedups in inference. The PR adds the `LLM_ARCH_EAGLE3` architecture, a dedicated encoder/decoder implementation (`src/models/eagle3.cpp`), feature extraction plumbing, GGML tensor sync support for GPU correctness, and EAGLE3 model conversion in `convert_hf_to_gguf.py`. Supports a wide range of models including Gemma 4, LLaMA 3.1/3.3, Qwen3, and GPT-OSS EAGLE3 speculators. Users can run with `--spec-type draft-eagle3` in `llama-server` or `llama-speculative-simple`.

**Affected areas:** speculative decoding, CUDA, ggml, server, python conversion

### [Backend] — PR #24306: vulkan: use medium matmul tile on Asahi Linux

**Merged:** 2026-06-11 | **Author:** @xingjianll

**Summary:** Switches the Vulkan matrix multiplication tile size from large to medium when running on Asahi Linux (Apple Silicon via the open-source Vulkan driver). Large tiles were found to cause GPU timeouts or performance regressions under the Asahi driver due to shader complexity limits. By conditionally selecting a medium tile layout when running on `eMesaHoneykrisp` (the Asahi driver name), matmul operations remain stable and performant. Asahi Linux users running llama.cpp through Vulkan will see more reliable inference without GPU hangs.

**Affected areas:** Vulkan, Asahi Linux, Apple Silicon
