### [Backend (OpenCL)] — PR #25537: opencl: add int8 dp4 dense and MoE prefill optimization for Adreno GPUs

**Merged:** 2026-07-11 | **Author:** @wanghqc (Qualcomm)

**Summary:** Adds a major int8 dot-product (DP4A) prefill path to the OpenCL backend for Qualcomm Adreno GPUs, alongside f16 GEMMs. Activations are quantized to q8_1 once and multiplied with weights (re-quantized to int8 if needed) using int8 dot-product instructions, covering q4_K, q6_K, q5_K, q8_0, q5_0, and IQ4_NL dense GEMMs. Additionally, dedicated MoE expert GEMM kernels with ragged tiling and MoE-combine fusion are introduced for q4_0 and MXFP4 MoE variants. Performance gains on the X2-90 Adreno are substantial: Qwen3-30B Q4_K sees +66.7% prefill speed, MXFP4 gets +54.5%, and Q4_0 gets +70.2%. Token generation speed is neutral (prefill optimization only). The feature is default-on for Adreno X2E but disabled on X1 GPUs where the f16 path remains faster due to narrower ALU resources. This is a major quality-of-life improvement for Qualcomm Snapdragon laptop users running local LLM inference.

**Affected areas:** OpenCL backend, Adreno GPU, int8 DP4A kernels, MoE prefill

### [Backend (Vulkan)] — PR #24877: Vulkan: route large matmuls to medium tile on Adreno

**Merged:** 2026-07-11 | **Author:** @Raman-Raje

**Summary:** Fixes a crash/hang on Qualcomm Adreno GPUs (Vulkan backend) during inference with q4_0 models when prompts grow past ~200 tokens. The root cause was that the large q4_0 matmul tile was being selected for long prompts, but that specific shader fails on Adreno hardware. The per-type shared memory guard did not catch this because the computed memory footprint (~16.9 KB) was under the reported `maxComputeSharedMemorySize`. The fix routes matmuls that would use the broken large tile to the medium tile instead when running on Adreno GPUs. Short prompts remain unaffected. This directly benefits mobile/Snapdragon users running local models through the Vulkan backend who previously experienced crashes or hangs on longer conversations.

**Affected areas:** Vulkan backend, Adreno GPU, matmul pipeline selection, q4_0

### [Server/API] — PR #25538: server: accept null sampling params

**Merged:** 2026-07-10 | **Author:** @ServeurpersoCom

**Summary:** Fixes the server's OpenAI-compatible API to properly accept `null` values for sampling parameters like `temperature`, `top_p`, etc. Previously, passing `"temperature": null` in a chat completions request would return a 400 error with a JSON type mismatch error. After this fix, null values are treated as "not specified" and fall back to the server's default sampling parameters, matching the OpenAI API specification. This is a follow-up to PR #24150 and resolves issue #25515. API users and client libraries that send null for optional parameters will now work correctly without needing to omit those fields entirely.

**Affected areas:** server, OpenAI-compatible API, JSON schema validation, sampling parameters
