### [Backend] — PR #23792: TP: quantized KV cache support

**Merged:** 2026-06-01 | **Author:** @JohannesGaessler

**Summary:** This PR removes the restriction that prevented using quantized KV cache types (e.g., Q8_0, Q4_0 for K/V) together with `--split-mode tensor` (tensor parallelism). The root cause was that flattening tensors for KV cache rotation lost shape information that the meta backend needed. The fix extends the `ggml_backend_meta_split_state` specification with a repeat count (`nr`) per segment, allowing the meta backend to correctly track data layout across flatten/reshape operations without changing the compute graphs. Users running multi-GPU tensor-parallel inference can now enable `--type-k` and `--type-v` quantization to reduce VRAM usage, which was previously blocked by an error message.

**Affected areas:** ggml-backend, llama-context

### [Backend] — PR #23989: Hexagon MUL_MAT, MUL_MAT_ID, FLASH_ATTN and GDN optimizations

**Merged:** 2026-06-02 | **Author:** @max-krasnyansky

**Summary:** A major update to the Qualcomm Hexagon (HTP/NPU) backend with substantial performance improvements across key operations. Adds F32 × F32 → F32 MUL_MAT support (needed for Gemma 4 Q4 GGUFs that contain F32 2D weights), reworks the MUL_MAT pipeline with resurrected pipelined/non-pipelined modes and unified quant/dequant helpers, adds MUL_MAT_ID support for MoE expert routing, improves FLASH_ATTN_EXT pipelining, and optimizes GATED_DELTA_NET with better DMA/VTCM usage. Benchmarks on Snapdragon devices show major gains — for example, OLMoE 1.7B prefill at pp1024 goes from 285 → 1263 t/s (~4.4× speedup), and Gemma 4 E2B at pp512 goes from 663 → 865 t/s. No user-facing flag changes.

**Affected areas:** Hexagon backend, ggml

### [Model Support] — PR #24060: Refactor `hparams.n_layer`

**Merged:** 2026-06-05 | **Author:** @ggerganov

**Summary:** A significant internal refactoring of how layer counts are managed in the hparams system. The old `n_layer` field is split into `n_layer_all` (total layers in the model file, including extra NextN/MTP layers) and a new `n_layer()` method (returns effective layers excluding NextN). This eliminates the `nextn_predict_layers` field and the `kv_only_nextn` boolean across dozens of model implementations (bailingmoe2, deepseek32, glm-dsa, glm4-moe, glm4, exaone4, exaone-moe, and more), replacing them with a consistent `n_layer_all` / `n_layer()` pattern. Also implements `set_recr_pattern()` which was previously a TODO stub. While this is a major internal cleanup with no direct user-facing changes, it paves the way for better MTP/NextN model handling and fixes potential layer counting bugs in hybrid architectures.

**Affected areas:** llama-hparams, all model implementations, kv-cache, memory, graph, model-loader

### [Bug Fix] — PR #23981: KV-cache SWA checkpoints store only non-masked cells

**Merged:** 2026-06-02 | **Author:** @ggerganov

**Summary:** Fixes issue #23720 where SWA (Sliding Window Attention) checkpoint restoration could fail, particularly with unified KV cache. The change reduces the size of SWA checkpoints by storing only non-masked cells, making it possible to always restore them correctly. This improves reliability for models using sliding window attention (Gemma 3, Cohere2, etc.) when saving/restoring KV cache state, especially in the server's continuous batching mode.
