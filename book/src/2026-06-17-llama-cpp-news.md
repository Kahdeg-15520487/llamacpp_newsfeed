### [Backend] — PR #22526: sycl: Add optional USM system allocations

**Merged:** 2026-06-17 | **Author:** @ifdu

**Summary:** Adds an optional feature for the SYCL backend to allocate large GPU buffers (≥1 GB) using USM (Unified Shared Memory) system allocations when supported by the device. When enabled via the `GGML_SYCL_USM_SYSTEM` environment variable, the system allocator manages memory migrations between host and device as needed, avoiding explicit copies. This is disabled by default and gracefully falls back to regular allocations if USM system allocations are not supported. SYCL users on compatible hardware (e.g., Intel GPUs with proper driver support) can enable this to reduce memory management overhead for large model weights.

### [Backend] — PR #24490: common: update logging to enforce max_capacity and optimize queue resizing

**Merged:** 2026-06-17 | **Author:** @max-krasnyansky

**Summary:** Fixes a logging queue unbounded growth issue where rapid logging could cause the queue to grow indefinitely until malloc fails. The logger now enforces a `max_capacity` limit of 4K entries. The queue resizing is rewritten so producer threads simply block on a full queue, while the consumer thread handles all resizing and flushing. This eliminates unnecessary copies of the current entry and during resizing since the queue is only modified by the consumer thread. This is a core infrastructure improvement that prevents out-of-memory crashes in high-throughput logging scenarios, particularly benefiting Hexagon backend Op tracing and any verbose debug logging.

**Affected areas:** common library, logging infrastructure, thread synchronization

### [Backend] — PR #24326: vulkan: record actual memory properties during buffer creation

**Merged:** 2026-06-17 | **Author:** @winstonma

**Summary:** Fixes a bug where Vulkan buffer `memory_property_flags` were mistakenly set to the requested flags rather than the flags actually allocated by the driver. This caused data corruption when the driver provided cached host-visible memory — the code would skip necessary cache flushes, thinking the memory was coherent. The fix records the actual memory properties from the allocation, ensuring `ggml_vk_buffer_write_2d` and `ggml_vk_buffer_read_2d` correctly detect `HostVisible` and `HostCoherent` states. Vulkan users on drivers that provide cached host-visible memory allocations will see improved data integrity.

**Affected areas:** Vulkan backend, buffer allocation, memory property tracking

### [Build/CI] — PR #24595: ci: fix vulkan docker images

**Merged:** 2026-06-17 | **Author:** @Kononnable

**Summary:** Fixes Vulkan Docker images that were broken since commit b9438 by running out of memory during shader generation in CI. The root cause was that `string_to_spv` uses `std::async` for parallel shader compilation and stores futures in a vector without ever releasing them. Each future allocates ~8MB of virtual memory for thread stacks, accumulating gigabytes across thousands of shaders and hitting system limits. The fix clears the futures vector periodically to release resources. Vulkan build maintainers using CI will now get correct Docker images again instead of silently corrupted artifacts.

**Affected areas:** CI pipeline, Vulkan shader compilation, Docker builds
