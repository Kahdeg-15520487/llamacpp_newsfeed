### [Server/API] — PR #23226: server + ui: SSE Replay Buffer

**Merged:** 2026-06-26 | **Author:** @ServeurpersoCom

**Summary:** Adds resumable SSE streaming to llama-server, keyed on conversation ID. When the HTTP socket drops (F5, tab close, iOS Safari backgrounding, transient network), generation continues server-side as a background task and bytes land in a bounded ring buffer. Clients can reattach via `GET /v1/stream/<conv_id>?from=N` to replay from any offset, list live sessions at `GET /v1/streams`, or stop with `DELETE /v1/stream/<conv_id>`. The Web UI snapshots live streams at mount and on visibilitychange, automatically reattaching mid-generation — even across full page refreshes. This is opt-in via `X-Conversation-Id` header on `POST /v1/chat/completions`; the OAI-compatible path is unchanged when absent. Router mode is also supported, enforcing one session per conversation across child instances.

**Affected areas:** llama-server, SSE streaming, Web UI, router mode, ring buffer

### [Backend/Performance] — PR #24426: CUDA: batch out_prod broadcast (dps2>1) path with cublasSgemmBatched

**Merged:** 2026-06-26 | **Author:** @leonardHONG

**Summary:** Optimizes the CUDA `out_prod` kernel's broadcast fallback path (where `dps2 > 1`, i.e., `src0` is broadcast along dim 2). Previously, this path issued a nested loop of individual `cublasSgemm` calls — one per slice. This PR replaces that with a single `cublasSgemmBatched` call using pointer arrays built by a small device kernel that handles `src0` broadcast indexing. Benchmarks on RTX 5090 show massive speedups: up to 282× for batch-size 256 (small GEMMs), 6.4× for mid-size GEMMs, and 1.35× even for large 1K GEMMs. No batch-count threshold is needed — the win holds down to `ne2 = 2`. All 71 `test-backend-ops OUT_PROD` tests pass.

**Affected areas:** CUDA backend, out_prod, cublasSgemmBatched, broadcast, RTX 5090

### [Bug Fix/Backend] — PR #24699: ggml-cpu: fix SVE leftover path in ggml_vec_dot_f32

**Merged:** 2026-06-26 | **Author:** @tdakhran

**Summary:** Fixes a correctness bug in the ARM SVE (Scalable Vector Extension) path of `ggml_vec_dot_f32` that caused 2D convolutions with kernel size 9 to produce wrong results on SVE-enabled ARM devices. The issue was that the SVE tail-handling code used `svmla_f32_z` (zeroing inactive lanes) instead of `svmla_f32_m` (merging/preserving inactive lanes), causing garbage data from inactive lanes to contaminate the dot product result. This is a correctness fix that affects all ARM devices with SVE support (e.g., Grace Hopper, certain Fujitsu A64FX, and newer ARM cores). A regression test was also added.

**Affected areas:** ARM SVE, CPU backend, ggml_vec_dot_f32, convolution, correctness

### [Bug Fix/Backend] — PR #24941: sycl: clamp softmax input to avoid underflow

**Merged:** 2026-06-26 | **Author:** @Jassieluo

**Summary:** Fixes a numerical stability bug in the SYCL softmax kernel where fully-masked rows (all `-INFINITY` inputs) produced `NaN` outputs due to `(-inf) - (-inf) = NaN` propagation through `sycl::native::exp(NaN)`. The fix clamps the softmax exponent input to a minimum of `-80.0f` using `sycl::max()`, which is safely above `ln(FLT_MIN) ≈ -87.33f` to avoid underflow to zero while preventing subnormal floating-point operations on the GPU. This is a two-line change that affects all SYCL users running models with masked attention, including BERT-style encoder models and any LLM with padding masks on Intel GPUs.

**Affected areas:** SYCL backend, softmax kernel, numerical stability, Intel GPU
